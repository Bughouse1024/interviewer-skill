# TechMap 分析器

## 任务

从学术论文、代码库、岗位描述等原材料中提取面试官的技术背景，构建 TechMap 知识库，用于推断其考察方向和出题模式。

## 提取维度

### 1. 岗位描述（JD）解析
- 核心职责：从 JD 提取 3-5 条核心职责，推断面试官最看重的能力
- 必要技能：JD 明确列出的硬技能要求
- 加分项：JD 中"优先"或"加分"的技能，推断面试官会如何区分候选人
- 隐藏考察点：从 JD 措辞推断未明说的考察意图

### 2. 研究方向与学术背景

- 教育经历：{学历} - {学校} - {专业}（如有多段经历按时间顺序列出，可跳过）
- 主要研究领域和细分方向
- 发表过的论文方向和核心贡献
- 惯用的研究方法论（理论推导？empirical 验证？ablation study？）
- 引用频率高的工作和研究

### 3. 技术栈与工程能力

- 惯用编程语言和框架
- 代码风格偏好（注释密度、模块化程度、命名习惯）
- 工程化程度（有无测试、CI/CD、文档习惯）
- 擅长的系统类型（模型框架？推理优化？）

### 4. 思路与偏好

- 面对问题时的思维框架（自顶向下？先跑实验再分析？）
- 偏好哪类解法（简洁优雅？工程鲁棒？理论最优？）
- LeetCode/Kaggle/论文中处理问题的习惯

### 5. 推断考察方向

- 基于技术背景，他最可能考察哪些领域
- 哪些话题在他的背景里是核心，哪些是边缘
- 他可能特别在意哪些细节（来自论文和代码中反复出现的模式）
- 哪些方向他几乎不会考（背景覆盖之外）

### 6. 出题模式推断

- 从已知题目归纳出题规律
- 他/她倾向于考原理还是应用？
- 是否喜欢给具体场景让候选人设计方案？
- 追问方向：追问细节？往 trade-off 问？往实际 case 延伸？

| JD 措辞 | 实际考察 |
|---------|---------|
| "有大规模数据处理经验" | 追问具体数据规模级别 |
| "具备良好的数理基础" | 概率统计等基础考察 |
| "熟悉量化研究流程" | 因子评价、回测等实操 |
| "模型训练经验" | XGB, LGM等机器学习模型 |

## 输出格式

```markdown
## 研究方向与学术背景
- 最高学历：{学历} - {学校} - {专业}
- 完整教育经历：{学历} - {学校} - {专业}（按时间顺序，可跳过）
- 主要研究方向：{xxx}
- 论文方向与核心贡献：{xxx}
- 方法论偏好：{理论推导 / empirical 验证 / ablation study 等}
- 高频引用工作：{xxx}

## 技术栈与工程能力
- 编程语言/框架：{xxx}
- 代码风格：{注释密度 / 模块化程度 / 命名习惯}
- 工程化程度：{有无测试 / 文档习惯等}
- 擅长系统类型：{xxx}

## 解题思路与偏好
- 思维框架：{自顶向下 / 先实验再分析 等}
- 解法偏好：{简洁优雅 / 工程鲁棒 / 理论最优 等}
- 权衡取舍态度：{是否喜欢追问 trade-off，倾向哪个方向}

## 推断考察方向
- 核心考察区：{xxx}
- 边缘区：{xxx}
- 特别在意的细节：{xxx}
- 几乎不会考的方向：{xxx}

## 出题模式
- 出题规律：{原理 / 应用 / 场景设计 等}
- 追问方向：{追问细节 / 权衡取舍 / 实际案例 等}
- 推断置信度：{高 / 中 / 低，注明依据}
```

## 注意事项

- 有直接证据的内容（论文、代码）优先于推断内容
- 重点提取"反复出现"的模式，而不是一次性偶发特征
- 技术背景的提取要服务于"他/她会怎么出题"这个核心问题，不要变成学术简历整理
- 如果材料不足，宁可留空，不要过度推断
- JD 是最直接的考察意图来源，优先级最高；论文和代码次之；纯推断最低
